# v0.7 Adoption Release

v0.7 is the adoption release. The goal is to show that
`repo-sentinel-lite` is useful as a portfolio hygiene gate, not just as an
isolated scanner demo.

## RSL-01 python-version-policy

Decision: choose scheme A.

`repo-sentinel-lite` supports Python 3.11 and newer. CI validates the package
on Python 3.11, 3.12, 3.13, and 3.14 across Ubuntu and Windows.

Scheme B, a Python 3.14-only experimental posture, is rejected for v0.7 because
the tool is meant for developer adoption. If a future release intentionally
returns to a 3.14-first posture, the README must say so in the first-screen
install area before the package metadata is narrowed.

## RSL-02 self-dogfooding

Dogfood `repo-sentinel-lite` in these portfolio repositories:

- `LogLens`
- `telemetry-lab`
- `sec-writeups-public`

Each dogfooding PR should add a small, reviewable hygiene gate rather than a
large security audit. The minimum integration contract is:

- install `repo-sentinel-lite` from production PyPI or a released tag
- run `repo-sentinel scan --fail-on-severity error .` in a repeatable local or
  CI path
- add `.reposentinel.toml` only when the target repository has intentional
  generated or fixture paths to ignore
- add `.reposentinel-baseline.json` only when existing findings are explicitly
  reviewed and accepted
- link to [`baseline-review.md`](baseline-review.md) when a baseline is added

## RSL-03 example-repo-before-after

Keep a small before-and-after example pair in this repository:

- `examples/dirty-repo` shows synthetic suspicious files, missing required
  files, and a redacted high-entropy finding.
- `examples/clean-repo` shows the same default rule surface after the standard
  repository files are present and no suspicious sample files remain.
- `examples/outputs/dirty-scan.json` and
  `examples/outputs/clean-scan.json` show deterministic scan output.
- `examples/outputs/dirty-baseline.json` and
  `examples/outputs/clean-baseline.json` show baseline output with high-entropy
  values redacted.
- `examples/outputs/dirty-fail-on-findings.txt` and
  `examples/outputs/clean-fail-on-findings.txt` show
  `--fail-on-findings` output and exit behavior.

These fixtures are intentionally tiny and synthetic. They should remain
reviewer-readable, deterministic, and safe to publish.

## RSL-04 pre-commit-integration-guide

Keep [`pre-commit-integration.md`](pre-commit-integration.md) as the adoption
guide for repositories that want `repo-sentinel-lite` as a hygiene gate.

The guide must cover:

- installing `pre-commit` and the local CLI
- configuring `repo-sentinel-error` and `repo-sentinel-warning`
- triggering and interpreting a failure
- creating `.reposentinel-baseline.json`
- reviewing baseline drift before committing it
- reusing the same gate in CI

## RSL-05 threat-model

Keep [`threat-model.md`](threat-model.md) as the public boundary document for
what `repo-sentinel-lite` does and does not prove.

The threat model must state that:

- `repo-sentinel-lite` is not a replacement for enterprise secret scanning
- a clean scan does not guarantee that a repository has no leaked secret or
  credential
- the scanner does not identify every credential format

## Review checklist

For each target repository, record:

- the integration entry point, such as CI, pre-commit, or a documented local
  command
- whether the first run passed without a baseline
- any baseline entries added, with a short explanation for each finding family
- whether `--fail-on-severity error` is the right first gate or whether the repo
  can safely use `--fail-on-severity warning`
- any project-specific ignore globs and why they are needed

## v0.7 readiness

v0.7 is ready when:

- RSL-01 remains true in package metadata, README, and CI
- RSL-03 example outputs continue to match the scanner behavior under tests
- RSL-04 pre-commit integration docs stay aligned with the provider manifest
- RSL-05 threat-model boundaries stay visible in README and reviewer evidence
- all three dogfooding targets have merged or intentionally documented
  integration outcomes
- any accepted baseline drift is reviewable and redacted
- the release notes describe adoption evidence separately from scanner feature
  changes
